# 学习工具改进工作流指南

> 本指南适用于持续改进 Claude Code 学习工具本身

---

## 核心原则

**先启动使用，边用边发现问题，再逐步改进**

不要一开始就建立复杂的工作流，而是**从简单开始，按需扩展**：

```
第1步: 直接开始使用工具
  └─> 发现问题，记录下来

第2步: 需要时再添加管理工具
  └─> 功能开关 / 分支 / 标签

第3步: 工具稳定后规范化
  └─> 完善工作流和文档
```

---

## 四个核心问题

### Q1: 记录时机 - 什么时候记录问题？

**原则**：发现即记录，不要依赖记忆

| 场景 | 时机 | 方式 |
|------|------|------|
| 使用中发现 bug | **立即** | 先在 IDEAS.md 快速记录 |
| 突然想到新功能 | **立即** | 哪怕只写关键词，防止忘记 |
| 使用体验不顺 | 用完后 | 有时间时整理记录 |
| 定期回顾 | 每周/每月 | 整理 IDEAS.md，归档已解决项 |

**为什么立即记录？**
- 工作记忆只能保持 5-9 项
- 5分钟后就可能忘记细节
- 好想法往往稍纵即逝

### Q2: 文件组织 - 需要几个文件？

**推荐：从简单开始，按需扩展**

**阶段 1：单文件模式（当前推荐）**
```
.claude/IDEAS.md  # 所有想法集中在一个文件
```

**阶段 2：多文件模式（当 IDEAS.md 超过 100 行时）**
```
.claude/

├── ARCHIVED_IDEAS.md     # 已解决的问题历史
└── FEATURE_FLAGS.md      # 功能开关配置（可选）
```

**阶段 3：完整模式（工具稳定后）**
```
.claude/
├── IDEAS.md              # 活跃想法
├── ARCHIVED_IDEAS.md     # 历史记录
├── FEATURE_FLAGS.md      # 功能开关
├── CHANGELOG.md          # 版本变更
└── ROADMAP.md            # 路线图（可选）
```

### Q3: Git 操作 - 什么时候用分支/标签？

**决策树**

```
发现问题
  │
  ├─> 小问题，5分钟能改好
  │     └─> 直接在 main 改，提交
  │
  ├─> 中等问题，需要10-30分钟
  │     └─> main 直接改，提交前确认
  │
  ├─> 大改动，可能影响现有功能
  │     └─> 创建 feature 分支
  │
  └─> 实验性，不确定是否保留
        └─> 创建 feature/experimental 分支
```

**标签使用时机**
```bash
# 重要里程碑
git tag v0.1-start      # 起点
git tag v0.2-usable     # 工具基本可用
git tag v0.5-stable     # 稳定版
git tag v1.0-release    # 正式发布
```

**分支使用时机**
```bash
# 功能开发
git checkout -b feature/问题修复

# 实验性功能
git checkout -b feature/experimental-新想法

# 完成后合并
git checkout main
git merge feature/问题修复
git branch -d feature/问题修复
```

### Q4: 是否需要创建专门的命令？

**建议：先手动，后自动化**

```
第1步：手动管理（1-2周）
  └─> 使用工具，收集真实需求
  └─> 了解哪些操作最频繁

第2步：评估需求
  └─> 如果某些操作反复执行 → 考虑自动化
  └─> 如果偶尔需要 → 手动处理即可

第3步：逐步添加命令
  └─> 先添加最常用的（如"记录问题"）
  └─> 根据使用情况增减
```

**不推荐一开始就创建完整命令系统的原因**：
1. 你还不知道哪些操作最频繁
2. 过早优化可能浪费精力
3. 真实需求可能与预期不同

## 问题记录格式

### 标准格式

**Bug 问题**
```markdown
## 🐛 [功能名称] 简短描述
- **发现时间**：YYYY-MM-DD
- **问题描述**：具体发生了什么
- **重现步骤**：
  1. 步骤1
  2. 步骤2
  3. 步骤3
- **期望行为**：希望怎么样
- **严重程度**：🔴 阻塞 / 🟡 困扰 / 🟢 轻微
- **状态**：待处理 / 进行中 / 已解决
```

**新功能想法**
```markdown
## 💡 [功能名称]
- **提出时间**：YYYY-MM-DD
- **想法描述**：
- **使用场景**：
- **实现思路**：（可选）
- **优先级**：P0 必需 / P1 想要 / P2 可有可无
- **状态**：待评估 / 计划中 / 进行中 / 已完成
```

**使用体验**
```markdown
## 📝 [体验项]
- **记录时间**：YYYY-MM-DD
- **描述**：
- **建议**：（可选）
```

### 快速记录格式（赶时间时用）

```markdown
## 2026-02-27 快速记录
- [ ] 进度显示不对
- [ ] 想要导出功能
- [ ] 命令太长
```

## 实用 Git 技巧

### 技巧 1：临时保存工作
```bash
# 改到一半需要切去做别的
git stash save "半成品功能X"
# ... 做别的事 ...
git stash pop  # 恢复工作
```

### 技巧 2：后悔药
```bash
# 改坏了，想撤销
git checkout -- 文件名      # 撤销单个文件
git reset --hard HEAD       # 撤销所有未提交的改动

# 提交错了，想回退
git revert HEAD             # 创建新提交撤销上一个
```

### 技巧 3：里程碑标记
```bash
# 工具可以用了
git tag v0.1-usable -m "工具基本可用"

# 某个功能完成了
git tag v0.2-stats -m "新增统计功能"
```

### 技巧 4：查看历史
```bash
# 查看最近修改
git log --oneline -10

# 查看某个文件的修改历史
git log --follow -- 文件名
```

## 改进流程示例

### 场景 1：发现 Bug
```bash
# 1. 记录问题到 IDEAS.md
# 2. 评估大小：小问题 → 直接改
# 3. 修改代码
# 4. 提交：git commit -m "fix: 修复XXX问题"
```

### 场景 2：添加新功能
```bash
# 1. 记录想法到 IDEAS.md
# 2. 评估影响：大改动 → 创建分支
git checkout -b feature/新功能
# 3. 开发完成后合并
git checkout main
git merge feature/新功能
# 4. 打标签
git tag v0.x.x-新功能
```

### 场景 3：实验性功能
```bash
# 1. 创建实验分支
git checkout -b feature/experimental-实验功能
# 2. 开发测试
# 3. 如果效果好 → 合并到 main
# 4. 如果不好用 → 直接删除分支
git branch -D feature/experimental-实验功能
```
